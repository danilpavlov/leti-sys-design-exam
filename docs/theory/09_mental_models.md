# Ментальные модели

## Что такое ментальная модель

Ментальная модель — это **внутреннее представление** человека о том, как что-то работает. Это «картинка в голове», которая помогает нам предсказывать поведение вещей, даже если мы не знаем, как они устроены внутри.

> Вы нажимаете выключатель — свет загорается. Вам не нужно знать про электрические цепи, провода и лампочки. У вас есть ментальная модель: «выключатель вверх = свет горит, выключатель вниз = свет не горит».

Ментальные модели есть у каждого человека для каждой вещи, с которой он взаимодействует. И для программных продуктов тоже.

---

## Три модели по Нормэну

Дональд Нормэн (Donald Norman), автор книги «Дизайн привычных вещей», выделяет три модели:

### 1. Модель пользователя (User's Model / Mental Model)

Как пользователь **думает**, что система работает. Эта модель формируется на основе:

- прошлого опыта с похожими системами,
- внешнего вида интерфейса,
- подсказок и текстов в интерфейсе,
- проб и ошибок.

> **Пример:** Пользователь думает, что «корзина» в интернет-магазине работает как корзина в реальном магазине: складываю товары, потом иду на кассу, могу передумать и убрать что-то обратно.

Модель пользователя всегда **упрощённая** и может быть **неправильной**. Пользователь не знает (и не должен знать), как система устроена внутри.

### 2. Модель разработчика (Designer's / Implementation Model)

Как система **на самом деле** устроена — её архитектура, база данных, алгоритмы, серверы.

> **Пример:** Разработчик знает, что «корзина» — это запись в таблице `carts` в PostgreSQL, связанная с `cart_items` через foreign key, а при оформлении заказа вызывается микросервис `order-service` через REST API.

Эта модель **полная и точная**, но **сложная**. Пользователю она не нужна и не интересна.

### 3. Модель интерфейса (System Image)

Как система **выглядит** и **ведёт себя** для пользователя. Это то, что пользователь видит: экраны, кнопки, текст, анимации, звуки, ошибки.

> **Пример:** Пользователь видит страницу с иконкой корзины, списком товаров, кнопкой «Удалить» напротив каждого товара, итоговой суммой и кнопкой «Оформить заказ».

Модель интерфейса — это **мост** между моделью разработчика и моделью пользователя. Это единственное, через что пользователь «общается» с системой.

## Главная задача UX-дизайна

!!! tip "Золотое правило"
    Сделать так, чтобы **модель интерфейса** формировала у пользователя **правильную ментальную модель**, максимально близкую к тому, как система реально работает.

Другими словами: все три модели должны быть **похожими**. Чем ближе модель пользователя к реальности — тем удобнее пользоваться системой.

### Когда модели совпадают — хорошо

Пользователь видит папки на рабочем столе и «понимает»: файлы лежат в папках, папки можно вкладывать друг в друга, файл можно перетащить из одной папки в другую. Его ментальная модель совпадает с тем, как это работает (хотя на самом деле никаких «папок» на диске нет — это абстракция файловой системы).

### Когда модели НЕ совпадают — проблемы

> **Пример:** Пользователь думает, что «Удалить из корзины» убирает товар насовсем. А на самом деле товар перемещается в «Отложенные». Пользователь заходит в корзину через неделю — а там снова товары. Он в шоке: «Я же удалил!»

> **Пример:** Пользователь думает, что кнопка «Сохранить» сохраняет данные на его компьютер. А на самом деле данные сохраняются в облако. Когда у него нет интернета — сохранение не работает, и пользователь не понимает, почему.

## Метафоры в интерфейсе

Метафора — это когда мы берём что-то знакомое из реального мира и используем как образ в интерфейсе. Метафоры помогают пользователю быстро построить правильную ментальную модель.

| Метафора | Откуда | Как используется |
|----------|--------|-----------------|
| Рабочий стол (Desktop) | Реальный рабочий стол | Файлы, папки, корзина |
| Корзина | Магазинная тележка | Сбор товаров перед покупкой |
| Закладки (Bookmarks) | Книжные закладки | Сохранённые страницы |
| Вырезать / Копировать / Вставить | Ножницы и клей | Работа с текстом |
| Облако (Cloud) | Небо / что-то далёкое | Хранение данных на сервере |
| Лента (Feed) | Газетная лента | Поток новостей / постов |
| Корзина (Trash) | Мусорное ведро | Удалённые файлы, можно восстановить |

!!! warning "Осторожно с метафорами"
    Метафора работает, пока она **точная**. Если реальная корзина работает не так, как корзина в магазине (например, товары исчезают через час), пользователь будет в замешательстве.

## Аффордансы (Affordances)

Аффорданс — это **подсказка в самом дизайне** о том, как объект можно использовать. Термин ввёл Джеймс Гибсон, а Дональд Нормэн популяризировал его в дизайне.

**Примеры из реального мира:**

- Дверная ручка «просит» потянуть, плоская пластина — толкнуть.
- Кнопка «выступает» — хочется нажать.
- Ползунок — хочется передвинуть.

**Примеры в интерфейсе:**

| Элемент | Аффорданс | Что подсказывает |
|---------|-----------|-----------------|
| Кнопка с тенью | «Выпуклая» — хочется нажать | Это можно нажать |
| Подчёркнутый синий текст | Ссылка | Это можно кликнуть |
| Полоска с тремя линиями (≡) | «Бургер-меню» | Тут скрыто меню |
| Поле с рамкой и курсором | Текстовое поле | Сюда можно ввести текст |
| Ползунок на шкале | Слайдер | Можно перетащить |
| Стрелки ◄ ► | Карусель / слайдер | Можно листать |

**Ложный аффорданс** — когда элемент выглядит кликабельным, но не является. Или наоборот — выглядит как текст, но это ссылка. Это раздражает пользователей.

## Когнитивная нагрузка

Чем сложнее ментальная модель, которую должен построить пользователь, тем выше **когнитивная нагрузка** — тем труднее пользоваться системой.

### Как снизить когнитивную нагрузку

| Что делать | Зачем |
|-----------|-------|
| Использовать знакомые метафоры | Пользователь «берёт» готовую ментальную модель |
| Быть консистентным | Не нужно строить новую модель для каждой страницы |
| Показывать состояние | Пользователь понимает, что происходит |
| Скрывать сложность | Показывать только то, что нужно сейчас |
| Давать обратную связь | Пользователь понимает, правильно ли он действует |
| Минимизировать запоминание | Показывать варианты, а не заставлять вводить наизусть |

## Примеры совпадения и несовпадения моделей

### Хороший пример: Перетаскивание файлов

Пользователь берёт файл мышкой и «перетаскивает» в папку. Ментальная модель: «Я переложил файл из одного места в другое» — как бумагу со стола в ящик. Модель совпадает с поведением системы.

### Хороший пример: Корзина покупок

Пользователь добавляет товары в корзину. Корзина показывает список и сумму. Можно убрать товар. Потом «идёшь на кассу» (оформление заказа). Всё как в реальном магазине.

### Плохой пример: «Сохранить» vs «Автосохранение»

Пользователь привык нажимать «Сохранить». Но в Google Docs всё сохраняется автоматически. Некоторые пользователи паникуют: «Где кнопка Сохранить? Мои данные потеряются!» — потому что их ментальная модель включает ручное сохранение.

Google Docs решил это, показывая текст «Все изменения сохранены» — чтобы скорректировать ментальную модель.

### Плохой пример: Версии документа

Пользователь думает: «Я отправил файл коллеге по email. Теперь у нас два независимых файла». В Google Docs модель другая: файл один, доступ общий. Если коллега изменит файл — изменения увидит и пользователь. Если не объяснить это, начнётся хаос.

## Итог

| Понятие | Простыми словами |
|---------|-----------------|
| Ментальная модель | «Картинка в голове» о том, как работает система |
| Модель пользователя | Как пользователь думает, что это работает |
| Модель разработчика | Как это работает на самом деле |
| Модель интерфейса | Как это выглядит для пользователя |
| Метафора | Знакомый образ для незнакомой вещи |
| Аффорданс | Подсказка в дизайне: «Нажми меня», «Перетащи меня» |
| Когнитивная нагрузка | Насколько тяжело голове «переварить» интерфейс |

!!! tip "Главный вывод"
    Хороший интерфейс — тот, который формирует **правильную** ментальную модель **быстро** и **без усилий**. Пользователь не должен думать о том, как пользоваться системой — он должен думать о своей задаче.
