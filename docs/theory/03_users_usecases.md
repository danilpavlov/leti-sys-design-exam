# Пользователи, Use Case, диаграммы

## Кто такие пользователи (Акторы)

Актор (Actor) — это кто-то или что-то, что взаимодействует с системой **извне**. Это не обязательно человек — это может быть другая система, устройство или сервис.

Акторов можно делить по:

- **опыту** — новичок, продвинутый, эксперт,
- **целям** — что хочет получить от системы,
- **правам доступа** — администратор, обычный пользователь, гость,
- **частоте использования** — ежедневно, раз в месяц, один раз.

### Примеры акторов

| Актор | Тип | Что делает |
|-------|-----|-----------|
| Покупатель | Человек | Ищет товар, оформляет заказ |
| Администратор | Человек | Управляет каталогом, пользователями |
| Платёжная система | Внешняя система | Обрабатывает оплату |
| Курьерская служба | Внешняя система | Получает данные для доставки |
| Таймер (Планировщик) | Система/время | Запускает ежедневную рассылку |

!!! warning "Важно"
    Актор — это **роль**, а не конкретный человек. Один человек может быть и покупателем, и администратором.

### Первичные и вторичные акторы

- **Первичный актор** — тот, кто **инициирует** взаимодействие. Например, покупатель начинает оформление заказа.
- **Вторичный актор** — тот, кого система **привлекает** для выполнения задачи. Например, платёжная система, к которой обращается наша система.

## Use Case простыми словами

Use Case (вариант использования) — это сценарий того, как актор достигает своей **цели** с помощью системы. Это не одно действие типа «нажать кнопку», а целый процесс от начала до конца.

!!! warning "Важно"
    Use Case — это **не кнопка** и **не экран**. Это **процесс**, в котором актор получает **ценный результат**.

### Примеры Use Case

| Use Case | Актор | Цель | Результат |
|----------|-------|------|-----------|
| Оформить заказ | Покупатель | Купить товар | Заказ создан, оплата прошла |
| Зарегистрироваться | Посетитель | Получить аккаунт | Аккаунт создан, можно войти |
| Найти товар | Покупатель | Найти нужную вещь | Список подходящих товаров |
| Отменить заказ | Покупатель | Отказаться от покупки | Заказ отменён, деньги вернут |
| Сформировать отчёт | Менеджер | Получить данные о продажах | Готовый отчёт (PDF/Excel) |

### Как назвать Use Case

Формула: **глагол + объект** (что делаем + с чем).

- Оформить заказ
- Добавить товар в каталог
- Восстановить пароль
- Просмотреть историю покупок

Плохие названия: «Заказ» (нет глагола), «Работа с товарами» (слишком абстрактно), «Нажать кнопку Купить» (слишком мелко, это не Use Case).

## Связи между Use Case

В UML есть три вида связей между Use Case:

### Include (включение)

Один Use Case **всегда** включает в себя другой. Без включённого Use Case основной не может работать.

> **Пример:** «Оформить заказ» **include** → «Подтвердить оплату» — нельзя оформить заказ без оплаты.

Используется, когда один и тот же шаг повторяется в нескольких Use Case (чтобы не дублировать).

### Extend (расширение)

Один Use Case **иногда** расширяет другой. Это необязательное поведение, которое срабатывает при определённом условии.

> **Пример:** «Оформить заказ» **extend** ← «Применить промокод» — промокод вводят не всегда, только если он есть.

### Generalization (обобщение)

Один Use Case — частный случай другого. Как наследование в программировании.

> **Пример:** «Оплатить заказ» — общий. «Оплатить картой» и «Оплатить через СБП» — частные случаи.

### Когда что использовать

| Связь | Когда | Ключевое слово |
|-------|-------|----------------|
| Include | Шаг обязателен и повторяется | «всегда» |
| Extend | Шаг необязателен, по условию | «иногда, если» |
| Generalization | Один вариант — частный случай другого | «является видом» |

## Use Case Diagram

Диаграмма Use Case — самая простая UML-диаграмма. Она показывает систему «с высоты»:

- **Акторы** — человечки (стикмены) снаружи.
- **Use Case** — овалы внутри прямоугольника.
- **Границы системы** — прямоугольник, внутри которого овалы.
- **Связи** — линии между акторами и Use Case.

### Что показывает диаграмма

- Какие пользователи есть у системы.
- Что система умеет делать (на верхнем уровне).
- Кто чем пользуется.
- Границы системы — что входит, а что нет.

### Чего НЕ показывает

- Порядок действий (для этого есть Activity Diagram).
- Детали реализации.
- Интерфейс.
- Структуру данных.

### Пример диаграммы (текстом)

```
┌──────────────────── Интернет-магазин ────────────────────┐
│                                                          │
│  (Найти товар)                                           │
│  (Оформить заказ) ──include──▶ (Подтвердить оплату)      │
│  (Просмотреть историю заказов)                           │
│  (Добавить товар в каталог)                              │
│  (Управлять пользователями)                              │
│                                                          │
└──────────────────────────────────────────────────────────┘

Покупатель ─── Найти товар
Покупатель ─── Оформить заказ
Покупатель ─── Просмотреть историю заказов
Администратор ─── Добавить товар в каталог
Администратор ─── Управлять пользователями
```

## Activity Diagram

Activity Diagram (диаграмма деятельности) показывает процесс пошагово: что происходит сначала, потом, какие есть развилки и параллельные действия.

### Основные элементы

| Элемент | Обозначение | Что значит |
|---------|-------------|-----------|
| Начало | Закрашенный кружок | Старт процесса |
| Конец | Кружок в кольце | Финиш процесса |
| Действие | Прямоугольник с закруглёнными углами | Один шаг |
| Решение (развилка) | Ромб | «Если… то…» |
| Слияние | Ромб | Объединение веток |
| Параллельность | Толстая горизонтальная линия | Действия идут одновременно |
| Дорожки (swimlanes) | Вертикальные полосы | Кто выполняет действие |

### Пример: Оформление заказа

```
[Начало]
    │
    ▼
[Выбрать товары]
    │
    ▼
[Перейти в корзину]
    │
    ▼
◇ Пользователь авторизован?
├── Да ──▶ [Выбрать способ доставки]
└── Нет ─▶ [Авторизоваться] ──▶ [Выбрать способ доставки]
                                        │
                                        ▼
                                [Ввести адрес]
                                        │
                                        ▼
                                [Оплатить заказ]
                                        │
                                        ▼
                              ◇ Оплата успешна?
                              ├── Да ──▶ [Показать подтверждение]
                              └── Нет ─▶ [Показать ошибку]
                                                │
                                              [Конец]
```

### Когда использовать Activity Diagram

- Для описания бизнес-процессов.
- Для детализации шагов Use Case.
- Когда нужно показать порядок действий, развилки и параллельные потоки.
- Для общения с заказчиком — диаграмма интуитивно понятна.
