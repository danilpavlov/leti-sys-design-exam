# Предметная область и Domain Model

## Что такое предметная область

Предметная область (domain) — это та часть реального мира, которую автоматизирует наша система. Грубо говоря, это «мир», в котором живёт наша программа.

Чтобы сделать хорошую систему, нужно **понимать предметную область**. Если мы делаем интернет-магазин, мы должны разбираться в том, как устроена торговля: что такое заказ, как работает доставка, что такое возврат товара.

> **Пример:** Для интернет-магазина предметная область — это товары, заказы, клиенты, доставка, оплата, возвраты, скидки.

> **Пример:** Для системы записи к врачу — это пациенты, врачи, расписание, приёмы, диагнозы, направления.

> **Пример:** Для банковского приложения — счета, переводы, карты, платежи, валюты, курсы.

!!! info "Зачем это нужно"
    Если разработчик не понимает предметную область, он будет делать систему «вслепую». Результат — система, которой неудобно пользоваться, потому что она не отражает реальные процессы.

---

## Методы описания предметной области

Есть несколько способов зафиксировать знания о предметной области:

### 1. Текстовое описание

Простое описание словами: что есть, как работает, какие правила действуют.

> «Клиент может иметь несколько заказов. Каждый заказ содержит один или более товаров. Заказ может быть отменён, если он ещё не отправлен.»

### 2. Глоссарий терминов

Словарь терминов предметной области. Очень полезная штука — чтобы все говорили на одном языке.

| Термин | Определение |
|--------|------------|
| Заказ | Запрос клиента на покупку одного или нескольких товаров |
| SKU | Уникальный идентификатор товарной позиции |
| Возврат | Процедура возврата товара и денег клиенту |
| Корзина | Временное хранилище товаров, выбранных для покупки |

!!! tip "Совет"
    Глоссарий нужно вести с самого начала проекта. Это избавляет от кучи недоразумений. Например, «клиент» и «пользователь» — это одно и то же или разное?

### 3. UML-диаграммы

Визуальные схемы — диаграммы классов, диаграммы состояний, диаграммы последовательностей. Подробнее ниже.

### 4. Схемы бизнес-процессов

Описание того, как работают процессы в организации: кто что делает, в каком порядке, какие документы передаёт.

## Domain Model (Модель предметной области)

Domain Model — это **визуальная модель** сущностей предметной области и связей между ними. Она показывает, какие «вещи» существуют в нашем мире и как они связаны.

!!! warning "Важно"
    Domain Model — это модель **реального мира**, а не базы данных. Здесь нет таблиц, первичных ключей и SQL. Только сущности и их отношения.

### Что входит в Domain Model

- **Сущности (entities)** — ключевые понятия предметной области (Клиент, Заказ, Товар).
- **Атрибуты** — свойства сущностей (Клиент имеет имя, email, телефон).
- **Связи (ассоциации)** — как сущности связаны (Клиент → делает → Заказ).
- **Кратность (multiplicity)** — сколько объектов с каждой стороны (один клиент — много заказов).

### Пример Domain Model: Интернет-магазин

```
┌──────────┐       делает        ┌──────────┐     содержит     ┌──────────┐
│  Клиент  │ 1 ───────────── * │  Заказ   │ 1 ──────────── * │ Позиция  │
│──────────│                    │──────────│                   │ заказа   │
│ имя      │                    │ номер    │                   │──────────│
│ email    │                    │ дата     │                   │ кол-во   │
│ телефон  │                    │ статус   │                   │ цена     │
│ адрес    │                    │ сумма    │                   └────┬─────┘
└──────────┘                    └──────────┘                        │
                                                                   │ *
                                                              относится к
                                                                   │ 1
                                                              ┌──────────┐
                                                              │  Товар   │
                                                              │──────────│
                                                              │ название │
                                                              │ цена     │
                                                              │ описание │
                                                              │ категория│
                                                              └──────────┘
```

Читаем так:

- Один клиент может делать много (*) заказов.
- Один заказ содержит много позиций.
- Каждая позиция относится к одному товару.

### Кратность (Multiplicity)

Кратность показывает, сколько объектов участвует в связи:

| Обозначение | Что значит | Пример |
|-------------|-----------|--------|
| `1` | Ровно один | У заказа ровно один клиент |
| `0..1` | Ноль или один | У клиента может быть (или не быть) бонусная карта |
| `*` или `0..*` | Ноль или много | У клиента может быть 0, 1, 2, ... заказов |
| `1..*` | Один или много | В заказе минимум одна позиция |

### Типы связей

| Тип связи | Что значит | Пример |
|-----------|-----------|--------|
| Ассоциация | Объекты знают друг о друге | Клиент — Заказ |
| Агрегация | «Содержит» (части могут существовать отдельно) | Кафедра — Преподаватель (преподаватель может перейти) |
| Композиция | «Состоит из» (части не существуют без целого) | Заказ — Позиция заказа (нет заказа — нет позиций) |
| Наследование | «Является видом» | Физическое лицо и Юрлицо — оба являются видами Клиента |

## Жизненный цикл сущности (State Machine)

State Machine (диаграмма состояний) показывает, через какие **состояния** проходит объект и что вызывает переход из одного состояния в другое.

### Зачем это нужно

Не все действия можно совершить с объектом в любой момент. Например, нельзя отправить заказ, который ещё не оплачен. State Machine помогает это формализовать.

### Основные элементы

| Элемент | Что значит |
|---------|-----------|
| Состояние (прямоугольник с закруглением) | В каком «режиме» сейчас объект |
| Переход (стрелка) | Смена состояния |
| Событие (надпись на стрелке) | Что вызвало переход |
| Начальное состояние (закрашенный кружок) | Стартовая точка |
| Конечное состояние (кружок в кольце) | Финал |

### Пример: Жизненный цикл заказа

```
[Начало]
    │
    ▼
┌─────────┐  оплата прошла   ┌──────────┐   отправлен    ┌────────────┐
│ Создан  │ ───────────────▶ │ Оплачен  │ ─────────────▶ │ Отправлен  │
└─────────┘                  └──────────┘                 └─────┬──────┘
    │                             │                             │
    │ отменён                     │ отменён                     │ доставлен
    │ клиентом                    │ клиентом                    ▼
    ▼                             ▼                       ┌────────────┐
┌──────────┐               ┌──────────┐                   │ Доставлен  │
│ Отменён  │               │ Возврат  │                   └─────┬──────┘
└──────────┘               └──────────┘                         │
                                                                │ возврат
                                                                ▼
                                                          ┌──────────┐
                                                          │ Возврат  │
                                                          └──────────┘
```

### Ещё пример: Жизненный цикл задачи (баг-трекер)

```
[Создана] → [В работе] → [На ревью] → [Тестирование] → [Закрыта]
                │                            │
                └──── Возвращена ◄────────────┘
```

### Правила State Machine

- Объект всегда находится **ровно в одном** состоянии.
- Переходы происходят по **событиям** (действиям пользователя, системным событиям, времени).
- Не все переходы возможны — нельзя перепрыгнуть из «Создан» сразу в «Доставлен».
- У каждого перехода может быть **условие** (guard) — например, «отменить можно только если заказ ещё не отправлен».

## Как строить Domain Model: пошаговый процесс

1. **Выписать все существительные** из описания предметной области — это кандидаты в сущности.
2. **Отфильтровать** — убрать синонимы, слишком абстрактные или слишком мелкие понятия.
3. **Найти атрибуты** — какие свойства есть у каждой сущности.
4. **Найти связи** — как сущности связаны друг с другом.
5. **Определить кратность** — один-к-одному, один-ко-многим, многие-ко-многим.
6. **Проверить с экспертом** — показать модель человеку из предметной области и спросить: «Правильно ли я понял?»

!!! tip "Совет"
    Не пытайтесь сделать идеальную модель с первого раза. Начните с основных сущностей и постепенно добавляйте детали.
