# Use Case: структура и ошибки

## Зачем описывать Use Case подробно

Диаграмма Use Case показывает **что** система умеет, но не показывает **как** это происходит шаг за шагом. Для этого каждый Use Case описывают текстом — в виде структурированного документа.

Подробное описание нужно для:

- разработчиков — чтобы понимать, что программировать,
- тестировщиков — чтобы знать, что проверять,
- заказчика — чтобы убедиться, что вы поняли его правильно.

---

## Структура Use Case

Полное описание Use Case обычно содержит такие разделы:

| Раздел | Что описывает | Пример |
|--------|--------------|--------|
| **Название** | Что делает (глагол + объект) | Оформить заказ |
| **Актор** | Кто инициирует | Покупатель |
| **Краткое описание** | В одном предложении — суть | Покупатель оформляет заказ на выбранные товары |
| **Предусловия** | Что должно быть ДО начала | Пользователь авторизован, в корзине есть товары |
| **Основной сценарий** | Шаги «счастливого пути» | 1. Открыть корзину → 2. Выбрать доставку → … |
| **Альтернативные сценарии** | Что если что-то пошло не так | Если оплата не прошла — показать ошибку |
| **Постусловия** | Что изменилось ПОСЛЕ выполнения | Заказ создан, на почту отправлено подтверждение |
| **Исключения** | Критические ситуации | Сервер платёжной системы недоступен |

---

## Полный пример Use Case

### Название: Оформить заказ

**Актор:** Покупатель

**Краткое описание:** Покупатель оформляет заказ на товары из корзины, выбирает способ доставки и оплачивает.

**Предусловия:**

- Покупатель авторизован в системе.
- В корзине есть хотя бы один товар.

**Основной сценарий (Happy Path):**

1. Покупатель открывает корзину.
2. Система показывает список товаров, количество и сумму.
3. Покупатель нажимает «Оформить заказ».
4. Система предлагает выбрать способ доставки (самовывоз, курьер, почта).
5. Покупатель выбирает доставку курьером и вводит адрес.
6. Система рассчитывает стоимость доставки и показывает итоговую сумму.
7. Покупатель выбирает способ оплаты (карта) и подтверждает.
8. Система отправляет запрос в платёжную систему.
9. Платёжная система подтверждает оплату.
10. Система создаёт заказ, присваивает номер и отправляет подтверждение на email.
11. Покупатель видит страницу «Заказ оформлен» с номером заказа.

**Альтернативные сценарии:**

- **4a. Покупатель выбирает самовывоз:** система показывает список пунктов выдачи, покупатель выбирает пункт. Переход к шагу 7.
- **7a. Покупатель вводит промокод:** система проверяет промокод, пересчитывает сумму. Если промокод неверный — показывает сообщение об ошибке.
- **9a. Оплата не прошла:** система показывает сообщение «Оплата отклонена» и предлагает попробовать снова или выбрать другой способ оплаты.

**Постусловия:**

- Заказ сохранён в системе со статусом «Оплачен».
- Покупатель получил email с подтверждением.
- Товары зарезервированы на складе.

**Исключения:**

- Платёжная система недоступна — показать сообщение «Сервис временно недоступен, попробуйте позже».
- Товар закончился на складе во время оформления — уведомить покупателя и предложить убрать товар из заказа.

---

## Основной сценарий vs альтернативный

- **Основной сценарий (Happy Path)** — когда всё идёт как надо, без ошибок и отклонений. Это самый типичный путь.
- **Альтернативный сценарий** — когда что-то идёт не по плану, но ситуация штатная (неверный пароль, промокод не подошёл, выбран другой способ доставки).
- **Исключение** — нештатная ситуация, которую система не может обработать нормально (сервер упал, база недоступна).

!!! tip "Как думать об альтернативных сценариях"
    На каждом шаге основного сценария спрашивайте себя: «А что если тут что-то пойдёт не так?» или «А какие ещё варианты у пользователя?»

## Уровни детализации Use Case

Use Case можно описывать с разной степенью подробности:

| Уровень | Что это | Когда используется |
|---------|---------|-------------------|
| **Brief (краткий)** | Одно предложение — суть | На ранних этапах, для общего обзора |
| **Casual (обычный)** | Несколько абзацев текста | Для обсуждения с заказчиком |
| **Fully dressed (полный)** | Все разделы по шаблону | Для разработки и тестирования |

**Brief пример:**
> Покупатель оформляет заказ на товары из корзины, оплачивает и получает подтверждение.

**Casual пример:**
> Покупатель открывает корзину, проверяет товары, выбирает доставку и оплачивает картой. Если оплата не проходит — может попробовать снова. После успешной оплаты получает email с номером заказа.

**Fully dressed** — это полный шаблон, как в примере выше.

## Частые ошибки студентов

| Ошибка | Почему это плохо | Как правильно |
|--------|-----------------|---------------|
| Описывают интерфейс вместо действий | Use Case — про процесс, не про кнопки | «Покупатель оформляет заказ», а не «Покупатель нажимает кнопку Купить» |
| Слишком мелкие Use Case | Теряется общая картина | «Оформить заказ», а не «Нажать кнопку Добавить в корзину» |
| Нет результата для пользователя | Use Case должен давать ценность | Результат: «Заказ создан», а не просто «Данные сохранены» |
| Нет альтернативных сценариев | Упущены важные ситуации | Всегда думать: «А что если тут ошибка?» |
| Технические детали в сценарии | Use Case — не код | «Система сохраняет заказ», а не «INSERT INTO orders...» |
| Нет предусловий | Непонятно, когда Use Case применим | Указывать, что должно быть до начала |
| Слишком абстрактно | Нельзя по этому тестировать | Конкретные шаги, а не «система обрабатывает данные» |

!!! tip "Главное правило"
    Use Case описывает **что происходит**, а не **как реализовано**. Пишите так, чтобы заказчик мог прочитать и сказать: «Да, так и должно работать».

## Советы по написанию

1. **Начинайте с основного сценария** — опишите «счастливый путь», а потом добавляйте альтернативы.
2. **Используйте активный залог** — «Система отправляет email», а не «Email отправляется системой».
3. **Нумеруйте шаги** — чтобы в альтернативных сценариях можно было ссылаться на номер шага.
4. **Чередуйте актора и систему** — шаг актора → реакция системы → шаг актора → …
5. **Не вдавайтесь в дизайн** — не пишите «в правом верхнем углу кнопка», пишите «пользователь подтверждает действие».
